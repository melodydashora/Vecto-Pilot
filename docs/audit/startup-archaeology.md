# Startup Archaeology

**Branch:** `startup-unification-pass` (off `5104fbf3` on `auth-hardening-pass`)
**Method:** `git blame -L <range>` per defensive line, then `git log -1 --format` on each unique introducing hash to recover subject, date, author, and body.
**Status:** Read-only. No edits to tracked files.

**Classification schema:**
- **LOAD-BEARING** — app cannot run without it (or its function); preserve in some form.
- **PAST-PATCH** — fixed a real bug whose root cause may now be gone; verify before deleting.
- **CARGO-CULT** — defensive code with no traceable trigger; safe to delete.
- **WRAPPER-ONLY-NA** — exists only because of a layer above (the wrapper, a removed optimization, etc.); dissolves automatically when that layer is removed.

### Step C Meta-Findings

**M8 — File-duality from convention-mismatch, not scar tissue.** The `.env` / `.env.local` co-existence isn't accumulated cargo-cult — it's an intentional structural twin maintained because `agent-server.js` (and 7 scripts) use `dotenv/config` (default = `.env`) while `gateway-server.js` uses custom `server/config/load-env.js` (loads `.env.local`). The duality emerged from two servers being authored under different env-loading conventions and never reconciled. The header comment in `.env` acknowledges this and prescribes manual structural-sync as the maintenance discipline. The only drift is the header's `.replit run` claim — `.replit:7` actually sources `.env.local`, not `.env`. **Refinement to authorship-trailer heuristic:** intentional design can also create structural duplication; check for self-documentation (file headers, README) before classifying as cargo-cult.

**M9 — Orphan-detection via zero-import grep is decisive for shadow-server questions.** The Phase 1 recon's `index.js` concern was deferred. Step C resolves it with three grep queries (imports, config refs, build refs) all returning zero. When a file is runnable (`node X.js`) but has zero importers AND zero config refs, it is orphaned by every meaningful definition of "in use." Caveat: callers outside the repo (production deploy scripts, ops runbooks) can't be detected this way — the verdict is "codebase-internally orphaned," not "globally unused."

**M10 — Catch-all proxy fallback = silent split-brain.** `server/agent/embed.js:151` makes the gateway PRETEND to fully serve `/agent/*` while actually depending on the standalone for legacy routes. In environments without the standalone (workspace, deployment), legacy routes return `ECONNREFUSED` (network error) instead of an explicit `503 AGENT_DISABLED` or `404`. This is a latent operational hazard — callers see intermittent network errors, not "feature unavailable." Phase 3+ should either inline the legacy routes into the embedded version OR explicitly disable the legacy surface in non-workflow environments. Not Manifesto-blocking.

## Banked Pattern: AI-Assistant Scar Tissue Accretion

**Diagnostic pattern:** A defensive code block is added by an AI assistant to fix a symptom. Within hours or days, a second defensive block is added by the same assistant to fix a symptom that the first block introduced (or that the first block partially papered over). Over time, the file accumulates layers of local defensive code, each rational on its own commit, none coherent in aggregate. The architecture-level question ("should this whole construct exist?") never gets asked because each commit is locally well-motivated.

**Decisive evidence:** Authorship-trailer archaeology. In Replit's commit pipeline, commits authored by the Replit Agent carry the trailer `Replit-Commit-Author: Agent` or `Replit-Commit-Author: Assistant`. Commits authored by Melody directly (or written by her with another AI tool like Claude Code) typically lack that trailer and use a different `git config user.name` casing (e.g., `Melody` vs the agent-typical `melodydashora`).

**In this archaeology:** 9 of 10 defensive ranges in `scripts/start-replit.js` were introduced by commits with `Replit-Commit-Author: Agent`. The one exception (`d24efc5fe`, by `Melody` directly) was a strategic refactor that fixed bugs IN the Agent-authored defensive code without revisiting whether the wrapper itself should exist. The Agent's defensive layer was 6.5 months old before the architectural question ("does a flagship rideshare app need a 216-line startup wrapper?") was finally asked.

**Generalization:** When auditing AI-assistant-authored infrastructure code, the authorship trailer is the first thing to check. If most of the defensive code carries `Replit-Commit-Author: Agent` (or equivalent) and the human-authored commits are limited to bug-fixing inside the defensive scaffolding, you are almost certainly looking at scar tissue accretion — and the right intervention is architectural deletion, not further patching.

**Cross-references in this repo:** This pattern likely repeats in `agent-server.js` (per Phase 1 recon §10 deferred files), `index.js` (the shadow-server question, also deferred), and possibly `server/bootstrap/*.js`. A grep along the lines of `git log --format="%H %an %s" --grep="Replit-Commit-Author: Agent" -- <file>` would surface the affected files quickly during Phase 3 expansion.
