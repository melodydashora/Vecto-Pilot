# Engineering Specification: Assistant Gateway and Operations Bridge

## 1. Scope

This specification defines the functional, operational, security, reliability, and non-functional requirements for the assistant gateway service, including health and readiness probes, request validation, multi-stage AI orchestration, policy loading, operational command forwarding, authentication checks, server binding, and timeout behavior.

## 2. System purpose

The assistant gateway shall:
- expose health and readiness endpoints
- accept assistant requests through a dedicated IDE-facing entrypoint
- validate inbound assistant payload invariants
- execute a multi-stage strategist → planner → validator model pipeline
- load assistant policy from disk
- enforce invariant checks on model output
- proxy a limited set of operational verbs to a separate agent service
- keep “assistant reasoning” and “agent execution” logically separated

---

## 3. Functional requirements

### 3.1 Application bootstrap

#### FR-BOOT-001
The system shall initialize an Express application.

#### FR-BOOT-002
The system shall enable JSON request parsing.

#### FR-BOOT-003
The JSON body parser shall enforce a request body limit of `5mb`.

#### FR-BOOT-004
The service shall bind using:
- `GATEWAY_PORT`, defaulting to `5000`
- `HOST`, defaulting to `0.0.0.0`

#### FR-BOOT-005
On successful startup, the service shall log the listening host and port.

#### FR-BOOT-006
The HTTP server shall set:
- `requestTimeout = 5000`
- `headersTimeout = 6000`
- `keepAliveTimeout = 5000`

---

### 3.2 Health and readiness endpoints

#### FR-HEALTH-001
The service shall expose a health endpoint at `/health`.

#### FR-HEALTH-002
The service shall expose a readiness endpoint at `/ready`.

#### FR-HEALTH-003
`GET /health` shall return HTTP `200` with body `OK`.

#### FR-HEALTH-004
`GET /ready` shall return HTTP `200` with body `READY`.

---

### 3.3 Environment and configuration inputs

#### FR-CONFIG-001
The service shall read configuration from environment variables.

#### FR-CONFIG-002
Recognized environment variables shall include:
- `EIDOLON_TOKEN`
- `AGENT_TOKEN`
- `EIDOLON_POLICY_PATH`
- `REPL_SLUG`
- `REPL_OWNER`

#### FR-CONFIG-003
`EIDOLON_POLICY_PATH` shall default to `config/eidolon-policy.json` when unset.

#### FR-CONFIG-004
`REPL_SLUG` and `REPL_OWNER` may default to empty strings when unset.

---

### 3.4 Error response utility

#### FR-FAIL-001
The service shall provide a standardized failure response helper.

#### FR-FAIL-002
Failure responses generated by the helper shall return JSON containing:
- `ok: false`
- `error: <message>`

#### FR-FAIL-003
The helper shall allow the caller to control HTTP status code and error message.

---

### 3.5 Request invariant validation

#### FR-INV-001
Assistant requests shall be validated before model execution begins.

#### FR-INV-002
The assistant request payload shall be an object.

#### FR-INV-003
The assistant request payload shall contain a `messages` field.

#### FR-INV-004
The `messages` field shall be an array.

#### FR-INV-005
If payload validation fails, the request shall be rejected with an invariant error.

---

### 3.6 Policy loading

#### FR-POL-001
The service shall load assistant policy content from a configured file path.

#### FR-POL-002
Policy loading shall use asynchronous file reads.

#### FR-POL-003
If the configured policy file cannot be loaded, the service shall return a fallback policy-missing marker string.

#### FR-POL-004
The fallback marker shall identify which policy target was requested.

---

### 3.7 Multi-stage AI orchestration

#### FR-AI-001
Assistant reply generation shall follow a single-path strategist → planner → validator pipeline.

#### FR-AI-002
The strategist stage shall call the `STRATEGY_CORE` model adapter.

#### FR-AI-003
The strategist stage shall receive:
- system policy text
- full message history

#### FR-AI-004
If the strategist stage fails, the request shall fail with a strategist error.

#### FR-AI-005
The planner stage shall call the `STRATEGY_TACTICAL` model adapter.

#### FR-AI-006
The planner stage shall receive a prompt derived from strategist output.

#### FR-AI-007
Planner output shall be parsed as JSON.

#### FR-AI-008
If the planner stage fails, the request shall fail with a planner error.

#### FR-AI-009
The validator stage shall call the `STRATEGY_CONTEXT` model adapter.

#### FR-AI-010
The validator stage shall receive a validation prompt derived from planner output.

#### FR-AI-011
If the validator stage fails, the request shall fail with a validator error.

#### FR-AI-012
Validator output should be parsed as JSON when possible.

#### FR-AI-013
If validator output is not valid JSON, the raw text shall be wrapped into an object form for downstream handling.

---

### 3.8 Prompt shaping

#### FR-PROMPT-001
The planner prompt shall convert strategist intent into exact steps and code edits.

#### FR-PROMPT-002
The validator prompt shall validate planner JSON, correct addresses, and normalize output.

#### FR-PROMPT-003
Prompt generation shall be deterministic from the prior stage’s output.

---

### 3.9 Output finalization and invariant enforcement

#### FR-FIN-001
The finalization stage shall enforce schema-strict invariant checks on validated output.

#### FR-FIN-002
The finalization stage shall reject invented or placeholder venue values.

#### FR-FIN-003
If final output contains disallowed venue placeholders such as `TBD`, `invented`, or `fake`, the request shall fail with `INVARIANT_NO_VENUE_INVENTION`.

#### FR-FIN-004
If invariant checks pass, validated output shall be returned unchanged.

---

### 3.10 Assistant response structure

#### FR-RESP-001
Successful assistant responses shall return JSON with `ok: true`.

#### FR-RESP-002
Successful assistant responses shall include an `identity` field.

#### FR-RESP-003
The identity string shall include:
- assistant name
- model identity label
- `REPL_SLUG`
- `REPL_OWNER`

#### FR-RESP-004
Successful assistant responses shall include a `triad` object containing:
- strategist output
- planner output
- validator output

#### FR-RESP-005
Successful assistant responses shall include a finalized `result` field.

---

### 3.11 Assistant IDE entrypoint

#### FR-ASST-001
The service shall expose the assistant entrypoint at `POST /assistant/*`.

#### FR-ASST-002
Assistant requests shall require `EIDOLON_TOKEN` to be configured.

#### FR-ASST-003
If `EIDOLON_TOKEN` is missing, the service shall reject assistant requests with HTTP `401` and `MISSING_EIDOLON_TOKEN`.

#### FR-ASST-004
Assistant requests shall validate request invariants before executing the model pipeline.

#### FR-ASST-005
If assistant execution fails, the service shall return HTTP `400` with:
- `ok: false`
- an error message from the thrown error, or
- `TRIAD_ERROR` as fallback

---

### 3.12 Operations bridge

#### FR-OPS-001
The service shall expose an operations endpoint at `POST /ops/:verb`.

#### FR-OPS-002
Operational requests shall require bearer-token authorization matching `AGENT_TOKEN`.

#### FR-OPS-003
If authorization does not match `Bearer ${AGENT_TOKEN}`, the service shall reject the request with HTTP `401` and `UNAUTHORIZED`.

#### FR-OPS-004
The operations bridge shall allow only the following verbs:
- `fs.read`
- `fs.write`
- `shell.exec`
- `sql.query`

#### FR-OPS-005
If the requested verb is not in the allowlist, the service shall reject the request with HTTP `403` and `VERB_NOT_ALLOWED`.

#### FR-OPS-006
Allowed operations shall be forwarded to `AGENT_BASE_URL + "/op/" + verb`.

#### FR-OPS-007
Forwarded operation requests shall:
- use method `POST`
- include bearer authorization using `AGENT_TOKEN`
- include `content-type: application/json`
- forward the request body as JSON

#### FR-OPS-008
The operations bridge shall relay the downstream agent’s status code.

#### FR-OPS-009
The operations bridge shall relay the downstream agent’s response body as text.

---

## 4. Security requirements

### 4.1 Assistant access control

#### SEC-001
Assistant request handling shall require assistant-token configuration before accepting work.

#### SEC-002
The service shall not process assistant requests when `EIDOLON_TOKEN` is absent.

### 4.2 Operational separation

#### SEC-003
Operational verbs shall remain separated from assistant reasoning paths.

#### SEC-004
The service shall preserve a distinction between “brain” functions and “hands” functions.

#### SEC-005
Operational requests shall not be permitted without valid agent authorization.

### 4.3 Verb restriction

#### SEC-006
Only explicitly allowlisted operational verbs shall be executable through the gateway.

#### SEC-007
The service shall reject all non-allowlisted operational verbs.

### 4.4 Output integrity

#### SEC-008
Final model output shall be checked for disallowed invented venue values before being returned.

#### SEC-009
The assistant pipeline shall fail rather than emit output that violates declared invariants.

---

## 5. Operational requirements

### 5.1 Health support

#### OPS-001
The service shall provide explicit health and readiness endpoints for orchestration and monitoring.

### 5.2 Model orchestration behavior

#### OPS-002
The assistant pipeline shall preserve full inbound message history for the strategist stage.

#### OPS-003
The assistant pipeline shall operate in a deterministic staged order:
1. strategist
2. planner
3. validator
4. finalization

### 5.3 Configuration behavior

#### OPS-004
The service shall support file-based policy configuration.

#### OPS-005
The service shall support deployment metadata injection through `REPL_SLUG` and `REPL_OWNER`.

### 5.4 Request forwarding

#### OPS-006
Operational requests shall proxy to an external agent service rather than executing tool actions directly in-process.

#### OPS-007
The gateway shall preserve downstream operation response codes and bodies.

---

## 6. Reliability requirements

### 6.1 Failure handling

#### REL-001
Invalid assistant payloads shall fail fast before model execution begins.

#### REL-002
Failure at any stage of the strategist, planner, or validator chain shall terminate the request with an error response.

#### REL-003
Policy-file read failures shall degrade to a deterministic fallback string rather than crash the service.

#### REL-004
Validator JSON parse failures shall degrade to wrapped text output rather than crash finalization immediately.

#### REL-005
Finalization invariant failures shall reject the request explicitly.

### 6.2 Timeout and hung-request control

#### REL-006
The service shall apply tight server timeouts to reduce risk of hung requests or hung probes.

#### REL-007
Timeout values shall remain low enough to support responsive failure behavior for stuck connections.

### 6.3 Downstream dependency behavior

#### REL-008
The operations bridge shall treat the external agent as the execution backend for allowed operational verbs.

#### REL-009
The gateway shall return downstream response status and body without requiring downstream-specific transformation.

---

## 7. Non-functional requirements

### 7.1 Maintainability

#### NFR-MAINT-001
The service shall keep model orchestration logic separated into discrete stages.

#### NFR-MAINT-002
Prompt-construction logic shall remain encapsulated in dedicated helper functions.

#### NFR-MAINT-003
Policy loading, finalization, and standardized error responses shall remain factored into reusable helpers.

### 7.2 Observability

#### NFR-OBS-001
The service shall log successful listener startup.

#### NFR-OBS-002
The identity response should expose deployment context through slug and owner fields.

### 7.3 Security posture

#### NFR-SEC-001
The service shall prefer explicit rejection over permissive fallback for invalid assistant requests.

#### NFR-SEC-002
The operations surface shall remain minimized through a fixed allowlist.

#### NFR-SEC-003
The assistant pipeline shall enforce content invariants before producing a success response.

### 7.4 Performance and responsiveness

#### NFR-PERF-001
The service should remain lightweight enough to answer health and readiness probes promptly.

#### NFR-PERF-002
Request parsing shall cap body size to limit oversized payload exposure.

#### NFR-PERF-003
Server timeouts should reduce the likelihood of indefinitely hung connections.

---

## 8. Required request sequence

The implementation shall follow this sequence for assistant requests:

1. receive `POST /assistant/*`
2. verify assistant-token configuration is present
3. parse and validate the JSON request body
4. enforce request invariants
5. load assistant policy from configured path
6. call strategist with policy and message history
7. generate planner prompt from strategist output
8. call planner
9. parse planner output as JSON
10. generate validator prompt from planner output
11. call validator
12. parse validator output as JSON when possible
13. finalize validated output and enforce invariants
14. return structured success response with identity, triad, and result

The implementation shall follow this sequence for operational requests:

1. receive `POST /ops/:verb`
2. verify bearer authorization against `AGENT_TOKEN`
3. verify the requested verb is allowlisted
4. forward the request to the external agent backend
5. relay downstream status code
6. relay downstream response body

---

## 9. Explicit prohibitions

The implementation shall not:
- process assistant requests when `EIDOLON_TOKEN` is missing
- accept assistant payloads that do not include a valid `messages` array
- skip staged strategist → planner → validator execution for assistant replies
- return success output that violates venue-invention invariants
- execute arbitrary operational verbs outside the explicit allowlist
- blur assistant reasoning and operational execution into the same endpoint path
- ignore downstream operation authorization requirements

---

## 10. Acceptance criteria

The system is compliant if all of the following are true:

- the service starts an Express server with JSON parsing capped at `5mb`
- `/health` returns `200 OK`
- `/ready` returns `200 READY`
- assistant requests are accepted only through `POST /assistant/*`
- assistant requests fail when `EIDOLON_TOKEN` is not configured
- assistant payloads require a `messages` array
- assistant execution follows strategist → planner → validator ordering
- strategist receives policy text and full message history
- planner and validator failures terminate the request with explicit errors
- invalid or placeholder invented venue output is rejected during finalization
- successful assistant responses include `ok`, `identity`, `triad`, and `result`
- operations requests are accepted only through `POST /ops/:verb`
- operations requests require valid bearer authorization using `AGENT_TOKEN`
- only `fs.read`, `fs.write`, `shell.exec`, and `sql.query` are permitted
- allowed operations are forwarded to the external agent backend and downstream status/body are preserved
- the server applies request, header, and keep-alive timeouts of 5000, 6000, and 5000 milliseconds respectively
