## 2026-05-07 (late evening): Coach payload-error fix + attachment options (Files / Folder / Camera)

Two related Coach UX gaps. Both client-side, no server changes.

### Finding A — Coach payload error (base64 inflation hits 10MB JSON limit)

`server/bootstrap/middleware.js:210` sets the `/api/chat` JSON body limit to **10MB**. Coach attachments encoded via `reader.readAsDataURL(file)` produce a base64 data URL that **inflates the payload by ~33%**. Trigger: a single 4K screenshot or two phone photos easily exceeded 10MB after base64 encoding, returning HTTP 413 with `code: 'payload_too_large'`. The client at `useCoachChat.ts:193-209` only special-cased `missing_timezone`; everything else fell through to a generic `"Sorry—chat failed: ..."` message. Driver hit this on cold-screenshot offers and didn't know what to do.

**Lesson 1 — Base64 inflation is a recurring class of "the limit you set is not the limit you have."** Whatever JSON body limit you configure, the actual content limit is ~75% of that for base64-encoded binary attachments. If the server says 10MB, drivers can effectively send 7.5MB of raw image data; anything more 413s. This is invisible in the limit definition itself and only surfaces at the failure threshold. Mitigation: either advertise the effective limit (compute and display "max 7.5MB images") or compress before encoding.

**Lesson 2 — The Coach LLM doesn't benefit from 4K image input.** Vision models like Gemini 3 internally downsample to ~768×768 or 1024×1024 before model input. Sending a 4MB photo to be downsampled to 800×800 by the cloud is pure cost (network, base64 inflation, JSON parse) for zero quality gain. Client-side compression to 2048px / JPEG quality 0.85 is essentially free win on payload size with zero loss for the actual use case. The cost is ~500ms-1s of compression on a mid-tier Android phone — visible enough to need a per-attachment loading state, fast enough not to be a UX problem.

**Lesson 3 — Specific error codes get specific UX, fallthroughs get generic text.** Pre-fix: `errData.code === 'payload_too_large'` fell through to `"Sorry—chat failed: ..."`. Post-fix: maps to `"Attachments are too large for the coach. Try a smaller image, take a new photo at lower resolution, or attach fewer files."` The driver knows what to do. Same pattern as the existing `missing_timezone` branch — it's the only error code with helpful UX, and we just expanded the list. Worth pinning as a convention: every server-side error code that has a clear user-action gets a corresponding client-side message.

**Lesson 4 — The 9MB precheck budget vs 10MB server limit.** Easy to set the precheck at 10MB and have the server still 413 because of message + thread history + snapshot + IDs eating into the limit beyond the attachment payload. The 1MB headroom is small but load-bearing. Worth pinning: when you have a server-side limit and a client-side precheck against the same number, they should differ by enough to absorb non-payload overhead.

### Finding B — Attachment options surface (Files / Folder / Camera)

Pre-fix: one Paperclip button → file picker. Post-fix: Paperclip button → popover with three options (Files / Folder / Camera), each routing to its own hidden input or modal.

**Lesson 5 — Mobile camera is one HTML attribute (`capture="environment"`).** The native camera UI on iOS and Android is opened by a regular `<input type="file" accept="image/*" capture="environment">`. No JavaScript needed for the mobile path; the OS handles capture. Desktop fallback uses `getUserMedia` + a canvas snapshot, but mobile is the dominant Coach use case (drivers in cars, taking dash/offer photos), so the simple attribute path covers ~80% of usage. Worth pinning: when implementing camera/file/microphone capture, check whether a plain HTML attribute already does what you need on mobile before reaching for `getUserMedia`.

**Lesson 6 — Mobile detection via UA-string is brittle on iPad.** iPad-in-desktop-mode (default on iPadOS 13+) reports as Mac in `navigator.userAgent`. Naïve `/iPhone|iPad|iPod|Android/i.test(...)` misroutes iPad users to the desktop modal even though their device has a real camera with a native picker. Feature-detect via `'ontouchstart' in window` correctly catches iPad-in-desktop-mode and any other touch-capable device. Always feature-detect over UA-sniffing when the feature decision matters.

**Lesson 7 — Folder pickers (`webkitdirectory`) return everything in the directory.** A driver picking their Downloads folder by accident could send 500 random files. The accept-list filter (only image/PDF/doc) plus a max-attachments cap (10) plus a count message ("3 unsupported skipped") is the guardrail. Without those three pieces, the folder option is a footgun.

**Lesson 8 — `getUserMedia` cleanup is privacy-load-bearing.** If a component unmounts (or modal closes) without calling `stream.getTracks().forEach(t => t.stop())`, the camera light stays on. This is one of the most user-visible privacy violations possible in a web app. The useEffect cleanup must include the explicit stop. Same pattern for microphone streams (Web Speech API's underlying MediaStream).

**Lesson 9 — `cancelled` flag in async start prevents close-during-init race.** If the modal opens and the user immediately closes it, `getUserMedia` could resolve into the void with the camera still running. The `cancelled` boolean closure pattern + check after await + close-tracks-if-cancelled handles this race. Pattern is general for any "open modal, async-fetch resource, render" component.

## 2026-05-07 (late): Falsy-temperature sweep + dead `users`-write deletion — code that lied about behavior

Two findings, both about *code that asserts something the runtime quietly ignores or contradicts.* Different from the earlier "silent-at-load-fires-at-runtime" forensic class (those were ReferenceErrors); these are "code looks correct, runtime semantics are wrong" cases.

### Finding A — Falsy-temperature trap (sister of falsy-coord, higher practical urgency)

`tempC ? convert : null` rejects `0°C` (freezing point, exactly 32°F). Same JavaScript-falsy class as the falsy-coord sweep (`32a6d472`), but with **higher practical urgency** — DFW gets sub-32°F multiple times per winter, so this bug fires for current users on cold mornings, unlike the lat/lng=0 case which has no current user impact.

Three direct conversion sites + ~12 display/filter/prompt sites swept. Display sites that emitted `'N/A'` or `'??'` for `0°` to LLM prompts were the most pernicious — silent miscommunication into Coach/consolidator prompt context. A driver opening the app on a freezing morning saw `Weather unavailable` in the UI and `"N/A°F"` in Coach reasoning prompts; the AI made cold-weather decisions without the cold-weather signal.

Already-correct shape coexisted at `consolidator.js:1241` (`!= null`), confirming the codebase knew the right pattern but hadn't applied it consistently. Same drift shape as the falsy-coord case.

**Lesson 1 — Falsy-zero is a recurring class, not a one-time bug.** Coordinates yesterday, temperatures today. Tomorrow it'll be percentages, accuracy values, surge multipliers, or anything else where `0` is a valid value but `!value` rejects it. The class is structural; it'll keep producing instances until either (a) TypeScript's strict-null-check forces explicit `lat: number | null` typing that makes `!lat` look wrong, (b) an ESLint rule flags `||` and `?:` shapes on numeric values, or (c) project culture shifts to `!= null` / `??` as the default. We've now done two sweep commits in three days; that's evidence the per-instance fix doesn't generalize on its own. Worth filing a follow-up: explicit project-wide convention against `||` substitution on numeric-typed values.

**Lesson 2 — `??` chains preserve `0`, `||` chains drop it.** `lat ?? defaultLat` returns `lat` even if `lat === 0`; `lat || defaultLat` returns `defaultLat`. Same JS-spec rule applied to temperature. The replacement was mechanical: `temperature || weather.temp || null` → `temperature ?? weather.temp ?? null`.

**Lesson 3 — LLM-prompt strings are the most invisible failure mode of this class.** A driver-facing UI showing "N/A°F" is at least visible. An LLM prompt containing `"Temperature: N/A°F"` produces *plausible-sounding strategy recommendations that ignore weather*. The Coach won't say "I don't know the temperature so I can't reason about it" — it'll output cold-weather-blind heuristics, and a driver on a 32°F morning will follow them without realizing the AI never had the signal. **Fix priority: prompt-prep sites > display sites > filter sites.** Prompt sites pollute downstream reasoning; display sites at least signal "data missing" to the user; filter sites just drop a row.

### Finding B — Dead `users`-table writes ("code lying about behavior")

`shared/schema.js:18` declares "users: SESSION TRACKING ONLY (Ephemeral) ... NO LOCATION DATA - all location goes to snapshots table." The live DB matches: `users` has 7 columns, zero of them location.

But `server/api/location/location.js:889-984` (pre-fix) had 13 location fields in two `db.update(users).set({...})` and `db.insert(users).values({...})` blocks. **Drizzle silently dropped every field that didn't match the schema definition.** So the actual SQL drizzle generated was: `UPDATE users SET updated_at = now() WHERE user_id = ?` and `INSERT INTO users (user_id, created_at, updated_at) VALUES (...)`. **Net runtime impact today: zero data leakage.** The DB and schema enforced the doctrine. **But the code was misleading by ~95 lines.** A future contributor reading the handler reasonably concludes "we write the active location to users on every resolve" and may build downstream logic on that false premise. Worse: someone could "fix" the apparent inconsistency by adding the missing columns to the schema, which would then activate the dead writes and contradict the doctrine without anyone noticing.

**Lesson 4 — Drizzle's silent-column-filtering is a feature with a UX cost.** When a schema definition doesn't include a column, Drizzle won't emit it in INSERT/UPDATE — and won't error. That's correct ORM behavior (lets you write code against the schema definition without fighting type errors), but it produces a class of bug where the source code says one thing and the runtime does another. This isn't unique to Drizzle — Prisma, TypeORM, and Sequelize all do similar filtering — but worth knowing. **Mitigation:** never trust the runtime SQL by reading the source's `.set({...})`. Read the schema definition for what actually lands. Even better: log the generated SQL during dev and verify the column list matches expectations.

Tonight's commit deletes the dead location-writes outright (~95 lines). The bigger architectural refactor — moving `session_id`, `current_snapshot_id`, `session_start_at`, `last_active_at` OUT of `users` to a new `active_context` table per Melody's target diagram — is filed at `docs/review-queue/PLAN_QUEUED_session-out-of-users-architecture-2026-05-08.md`.

## 2026-05-07: Undefined `reqId` in `/snapshot` full-mode validation error path — second instance of silent-load runtime-fires identifier class

- **Symptom:** `POST /api/location/snapshot` with a malformed full-mode SnapshotV1 body (`coord` set, but missing `resolved.timezone` or `coord.lat/lng` or `snapshot_id`) crashed with `ReferenceError: reqId is not defined` at `location.js:1718, 1720`. The intended client-facing error contract was never reached. Live curl confirmed pre-fix: `{"ok":false,"error":"snapshot_failed","message":"reqId is not defined","req_id":"edcd5a4a-..."}`.
- **Root cause:** The handler at `location.js:1549` referenced `reqId` at lines 1718 and 1720 without ever declaring it in the function scope. Almost certainly a copy-paste from `server/api/location/snapshot.js:35`, which DOES declare `const reqId = crypto.randomUUID()` at the top. The declaration line was skipped during the copy; the references survived. JavaScript's lexical resolution then fell through to global scope, found nothing, threw at runtime when the validation-failure path executed.
- **Trigger condition:** Specifically the full-mode-with-invalid-payload path. Minimal-mode (flat `lat`/`lng` only) routes to a different branch and isn't affected. The web client never produces malformed full-mode bodies, so this only fires when a curl/test/scraper sends a partial SnapshotV1 — making the bug invisible to UI-level smoke testing.
- **Forensic class — second occurrence in one week:** Same shape as the `thinkingLevel = high` ReferenceError fixed in commit `daa430f3` (Coach was broken because the gemini-adapter default-value expression referenced an undeclared `high` identifier). Both bugs:
  - Lint-clean, typecheck-clean, build-clean.
  - Module loads without error.
  - Server boots fine.
  - Only fire at runtime when a specific code path executes — and the specific path is one that production tests rarely exercise (validation-failure paths, default-parameter-fallback paths).
  - The static checkers can't catch identifier references that don't resolve, because the file *parses* fine. JavaScript doesn't know the identifier is undefined until the runtime tries to look it up in a scope chain and fails.
- **Lesson 1 — Naming symmetry hides typos.** `httpError`'s 5th parameter is named `reqId`; the call site does `httpError(res, ..., reqId, ...)`. The two `reqId`s match by name, so a code reviewer reads "this call passes `reqId` to the `reqId` parameter" and stops checking whether the *local* `reqId` actually exists. The asymmetry that would have prompted "wait, what is this thing?" never appeared. Moral: when a function's parameter name matches the local variable name a caller is likely to pass, the typo where the caller's variable is undeclared becomes structurally hard to spot. Watch for it specifically when copy-pasting call sites between handlers.

- **Lesson 3 — Validation-failure paths need integration tests, not just happy-path smoke.** Both bugs hid in code that only runs when the input fails some validator. The web client never produces those inputs. UI smoke tests therefore couldn't catch either bug. The fix-cost of one curl test per error path (parametrized by which field is missing) is negligible; the audit cost of *not* having that coverage is catching the bug in production logs after a real client trips it. Worth folding into integration test policy.
- **Lesson 4 — The `message` field of error responses bleeds implementation detail.** The pre-fix outer catch wrapped the ReferenceError into `{ ok: false, error: 'snapshot_failed', message: 'reqId is not defined' }`. That `message` reveals an internal variable name and suggests the existence of a bug to anyone who triggers the path. Information disclosure on the error path is small but non-zero — an attacker probing endpoints learns "this server has an undefined identifier in error handling code," which is more diagnostic info than they should get. Generic outer-catch error messages should be opaque (`'internal_error'`) and detailed messages should only land in server-side logs. Out of scope for this fix; flag as a hardening follow-up.
- **Resolution:** Added two lines at the top of the `POST /snapshot` handler:
  ```javascript
  const reqId = crypto.randomUUID();
  res.setHeader('x-req-id', reqId);
  ```
  The first line gives the references at lines 1718 and 1720 a same-scope binding to resolve to. The second line mirrors `snapshot.js:36` — exposes the identifier to clients via `x-req-id` header for support-side correlation. Two reasons to pick option (a) "declare reqId" over option (b) "rewrite refs to use cid": (1) the response field is literally named `req_id` and should carry a per-request value, not a correlation value; (2) `cid` can be reused if the client sends the same `x-correlation-id` on retries, while a per-handler `randomUUID()` cannot collide.

## 2026-05-07: Snapshot ownership gaps — global `requireAuth` masking missing per-route `requireSnapshotOwnership`

- **Symptom:** Two `:snapshotId`-bearing routes in `server/api/location/location.js` were gated by `requireAuth` (via `router.use(requireAuth)` at line 30) but **not** by `requireSnapshotOwnership`. The two routes:
  1. `PATCH /api/location/snapshot/:snapshotId/enrich` (location.js:2294) — LIVE, called from `client/src/contexts/location-context-clean.tsx:594` to update `weather`/`air` on the user's snapshot. With only `requireAuth`, any authenticated User B knowing User A's snapshot UUID could overwrite A's `weather`/`air` and trigger the readiness gate at line 2341 to flip A's `status` from `pending` to `ok` (or vice versa) with poisoned values, cascading into A's briefing → strategy → venues pipeline.
  2. `GET /api/location/snapshots/:snapshotId` (location.js:2376, plural) — DEAD route (zero callers verified across server/client/scripts/tests). With only `requireAuth`, any authenticated user knowing a snapshot UUID could fetch the full row of any other user's snapshot — full IDOR-read.
- **Severity classification:** R1 is HIGH by structural taxonomy (cross-user write). Practical urgency was moderated by the fact that the leading enabling vector — the cross-user snapshot-id leak in `/resolve`'s reuse path — was already closed in commit `be8f122e` (2026-05-06). R2 was a latent-but-dead IDOR — taxonomy-HIGH if exploited, zero callers in practice.
- **Root cause:** `location.js` applies `requireAuth` globally via `router.use(requireAuth)`. This is sufficient for routes that don't take a user-scoped resource ID. The moment a route takes `:snapshotId`, the global `requireAuth` becomes insufficient — it confirms *some* user is logged in but not that they own the resource. Sibling routers (`briefing.js`, `snapshot.js`) apply both middlewares per-route inline (`requireAuth, requireSnapshotOwnership`), making missing ownership checks visually obvious during code review. In `location.js`, the per-route omission was invisible: every route declaration looked structurally identical (only the path string varied), so the eye scanned past the missing layer.
- **Lesson 1 — `requireAuth` is the wrong question for any route taking a user-scoped resource ID.** The right question is "given this authenticated user, may they touch *this specific* resource?" — which is exactly what `requireSnapshotOwnership` answers and `requireAuth` doesn't. The audit pattern: any route declaration that takes `:snapshotId` (or any `:user_id`-scoped param) without `requireSnapshotOwnership` (or its equivalent) is presumed-vulnerable until proven otherwise. The presumption-of-guilt is structural; reverse it and audits miss things.
- **Lesson 2 — Mixing middleware-application styles in the same router is an audit-discipline hazard.** When `briefing.js` writes `requireAuth, requireSnapshotOwnership` per-route, a missing `requireSnapshotOwnership` jumps off the page on visual scan. When `location.js` hides `requireAuth` in a `router.use()` higher up, the per-route declarations all look identical and the eye stops noticing what's missing. Two structural fixes (either one alone is sufficient): (a) audit-time grep that flags any `:snapshotId` route lacking `requireSnapshotOwnership`, run as part of CI; (b) move `location.js` to per-route auth so visual scanning surfaces gaps the same way it does in `briefing.js`. This commit takes (b) implicitly for the affected route — the enrich route now declares both gates inline, even though `requireAuth` is technically redundant given the `router.use()` — because making the audit signal explicit at the declaration outweighs the diff cost.
- **Lesson 3 — Dead routes are the most underweighted security class.** `GET /snapshots/:snapshotId` had zero callers, which makes it feel like "not a real bug." But every dead route is one careless `fetch()` away from being live, and the contributor adding that `fetch()` won't know the route bypasses ownership protection — they'll see the route's existence as evidence it's safe. Same anti-pattern as `shared/identity.ts` (deleted in commit `b8a6b637`). The clean root-cause fix is **deletion**, not gating. Gate-not-delete leaves a duplicate of an already-properly-gated canonical route (`/api/snapshot/:snapshotId` at `snapshot.js:220`) sitting in the codebase as drift bait. Dead duplicate code that contradicts the security model is a regression vector by construction.
- **Lesson 4 — Existence checks duplicated across handler and middleware are smells.** Pre-fix, the enrich handler did its own `SELECT snapshot_id FROM snapshots WHERE snapshot_id = ?` to verify existence (lines 2304-2312). After applying `requireSnapshotOwnership`, that query becomes redundant — the middleware already does the lookup, with stricter semantics (it also enforces ownership) and attaches the result to `req.snapshot`. When you add a middleware that does X, delete the handler's duplicated X. Two queries that should be one is the same anti-pattern as two implementations of the same identity rule (e.g., the `existingUser.user_id` vs `userId` split that produced the cross-user `/resolve` bug, fixed 2026-05-06).
- **Resolution:**
  1. R1: Added `requireSnapshotOwnership` to `PATCH /snapshot/:snapshotId/enrich`. Removed the redundant existence query (lines 2304-2312), the empty-snapshotId guard (lines 2299-2301), and the "Verify snapshot exists" comment block — middleware handles all three.
  2. R2: Deleted `GET /snapshots/:snapshotId` (plural) entirely. The canonical singular `GET /api/snapshot/:snapshotId` route at `snapshot.js:220` already serves this need and is properly gated.
  3. Added `requireSnapshotOwnership` import to `location.js`.
  4. Filed `claude_memory` audit row, status `resolved` on landing.
- **Files (this commit):** `server/api/location/location.js`, `LESSONS_LEARNED.md`.
- **Plan:** `docs/review-queue/PLAN_snapshot-ownership-gaps-2026-05-07.md`
- **CLAUDE.md ref:** Rule 1 (planning before implementation), Rule 9 (zero-tolerance drift), Rule 16 (auth-keyed identity).
- **Adjacent routes deferred to a future pass:** `POST /api/location/release-snapshot` (location.js:166) and `POST /api/location/snapshot` (location.js:1548). Neither takes `:snapshotId` in the URL, so they're outside this scope, but if either reads `snapshotId` from the body or operates on `current_snapshot_id` they may share the audit-blindness shape. Flagged for next-pass review.


## 2026-05-06: Cross-user `/resolve` corruption — non-unique column used as auth-route lookup key

- **Symptom:** When a user authenticated on a browser whose stored `device_id` matched a `users` row belonging to a *different* user, `GET /api/location/resolve` updated the wrong row. User A's location/coord_key/formatted_address/city/state/timezone got overwritten with User B's data; the response carried User B's `user_id` while the SQL UPDATE targeted User A's `user_id` (`server/api/location/location.js:970` vs lines 913/1086/1250 pre-fix). A snapshot-id leak ran in parallel — the snapshot reuse path (`existingUser.current_snapshot_id`) returned A's snapshot UUID to B (bounded by `requireSnapshotOwnership` to information-disclosure-plus-broken-UX, not full IDOR).
- **Root cause:** The `users` table column `device_id` was `notNull()` but **not UNIQUE** (`shared/schema.js:21`, pre-removal). The route's existing-user lookup keyed on `device_id` alone (`findFirst where eq(users.device_id, deviceId)`), with a `findFirst-by-user_id` fallback that ran ONLY when the device lookup returned null. So any device hit short-circuited the auth-bound lookup and the route silently bound the request to the wrong row. The "If authenticated user differs from device's current user, update the user_id" log line at the original line 916-918 read as if it linked the device to the new user — but the conditional body was a single `console.log` call. The log was the entire implementation.
- **Amplifier (dev):** `location.js` had a fallback that synthesized `dev-admin-{lat6d}_{lng6d}` device_ids when none arrived in the query. Two test users at the same coordinates collided on device_id, making the bug trivially reproducible in dev/E2E with no adversarial intent.
- **Lesson 1 — When `requireAuth` runs upstream of a route, identity lookups MUST key on `req.auth.userId`.** Any other key is at best redundant and at worst a cross-user data-corruption vector. The original lookup keyed on `device_id` because the route predated `requireAuth`; once `requireAuth` was added (2026-02-12), the device-keyed lookup became architecturally inconsistent with the auth model — but no audit caught it because the old logic kept "working" for the single-user-per-browser case.
- **Lesson 2 — `notNull()` is not `unique()`.** Schema reading habit: every `notNull` column you encounter prompts the question "is this 1:1 or 1:N with its parent?" `users.device_id` read like "the device this user signed up on" (1:1) but the schema actually allowed 1:N, and that's where the exploit lived. If a column is going to be used as a lookup key under any code path, it must be `unique()` or the lookup must be paired with a more specific filter.
- **Lesson 3 — A consistency invariant the author intended but never enforced is a smoking gun.** The split between `WHERE user_id = existingUser.user_id` (line 970) and `WHERE user_id = userId` (lines 1086, 1250) showed the original author *believed* `existingUser.user_id === authenticatedUserId` but never expressed that as code. When the same logical identity is referenced two different ways within the same function, the function is one input away from a divergence.
- **Lesson 4 — Information-disclosure leaks bound by downstream middleware are still bugs.** `requireSnapshotOwnership` (`server/middleware/require-snapshot-ownership.js:53`) prevented full IDOR on the leaked snapshot UUID by enforcing `snapshot.user_id === req.auth.userId` on every read route — but the leak still exposed UUIDs and broke User B's UX (their client tried to fetch A's snapshot and got 404). A "the harm is bounded by the next middleware" defense is fragile by construction; fix the leak at source, don't rely on the gate.
- **Lesson 5 — When auth becomes the identity primitive, vestigial identity columns must be removed, not just stop being read.** The `users.device_id` column had no remaining purpose once the lookup was rewritten — but leaving the column in place would have left a `notNull` field that every INSERT had to fabricate placeholders for (`web-${randomUUID()}` callsites in `auth.js`), and the next contributor reading the schema would re-introduce the same lookup pattern that caused the bug. The fix is to drop the column structurally so the original mistake cannot be made again.
- **Resolution:**
  1. Refactored `/resolve` user lookup to a single `findFirst where eq(users.user_id, authenticatedUserId)` — no more device-keyed path. UPDATE WHERE collapses to the same `authenticatedUserId` as the rest of the function.
  2. Dropped `device_id` from `users`, `snapshots`, `discovered_traffic` (waterfall e2e). Migration: `migrations/20260506_drop_device_id_from_users_snapshots_traffic.sql`.
  3. Preserved `device_id` in Siri Shortcuts pipeline (`offer_intelligence`, `intercepted_signals`, `/api/hooks/*`) — Siri cannot send JWT tokens, so device_id remains the auth primitive on those endpoints only.
  4. Cascade-cleaned every `users.device_id` / `snapshots.device_id` reader and writer across server, client, validation schemas, tests, and storage keys. Stale comments referring to removed code were deleted in the same commits (no `// REMOVED on YYYY-MM-DD` breadcrumbs in live source — the migration file and commit message carry the historical record).
- **Files (cleanup PR, 2026-05-06):** `shared/schema.js`, `server/api/location/location.js` (primary refactor), `server/api/auth/auth.js` (6 INSERT callsites), `server/api/strategy/strategy.js`, `server/api/location/snapshot.js`, `server/lib/briefing/pipelines/traffic.js`, `server/scripts/self-healing-monitor.js`, `server/logger/workflow.js`, `server/validation/schemas.js`, `client/src/contexts/location-context-clean.tsx`, `client/src/components/GlobalHeader.tsx`, `client/src/constants/storageKeys.ts`, `client/src/constants/apiRoutes.ts`, `tests/triad/test-pipeline.js`, `migrations/20260506_drop_device_id_from_users_snapshots_traffic.sql`.

## 2026-05-03: AUTH-003 — Standard JWT Migration with Dual-Verify Dispatch

- **Symptom (pre-migration):** `docs/architecture/AUTH.md` §2 documented the auth token as a custom 2-segment `userId.HMAC-SHA256(userId, secret)` despite the env var being named `JWT_SECRET`. No `exp`/`iat`/`iss`/`aud` claims; token-level expiry only enforced via server-side session TTL. Third-party integrations couldn't verify our tokens without holding the signing secret. AUTH.md §9 TODO #3 had been open since 2026-04-10.
- **Recon discovery #1 — duplicate ARCH-001 ID across docs.** `MASTER_ROADMAP.md:12` labeled the JWT migration ARCH-001; `DOC_DISCREPANCIES.md:98` already used ARCH-001 for the (legitimate, separate) session-table-split tech debt. Resolved by relabeling the JWT migration to AUTH-003 (derived from AUTH.md §9 TODO #3) — the session-split kept ARCH-001 since it was the older claim with valid tracker provenance.
- **Recon discovery #2 — 53-line duplicate HMAC verifier hiding in `server/api/location/location.js:441-493`.** A full inline copy of token-split + HMAC-recompute, doubly redundant: (a) the route already had `router.use(requireAuth)` at line 30, so `req.auth.userId` was already populated; (b) it only knew HMAC, so it would have silently rejected new JWT tokens post-migration. Exact "duplicate logic = bug" violation per CLAUDE.md Rule 9. Consolidation collapsed 53 lines into `const authenticatedUserId = req.auth?.userId || null;`.
- **Recon discovery #3 — `jose` v6.1.1 already a dependency.** No new package needed. Existing `server/lib/jwt.ts` was a 14-line dev stub with zero importers — replaced with real `signJWT`/`verifyJWT` helpers.
- **Implementation — dual-verify dispatch by segment count.** `verifyAppToken(token)` routes 3-segment input to `verifyJWT` (jose-based, HS256, claims pinned via `algorithms: [HS256]` to defend against algorithm-confusion attacks), and 2-segment input to `verifyLegacyHMAC` (kept temporarily for transition). NO fall-through between paths — each segment count maps to exactly one verifier — preventing downgrade attacks where an attacker strips a segment to bypass JWT validation. `matrixLog [AUTH] [LEGACY_HMAC_USED]` per-call counter signals when the legacy traffic drains so Phase 1.5 cleanup PR can safely delete the legacy branch.
- **Test layer:** 26 new unit tests across `tests/auth/jwt-helpers.test.js` (16 cases: round-trip, signature/payload tampering, exp/iat/iss/aud rejection, algorithm-confusion attack, missing-sub, malformed, input validation, immutable JWT_CONFIG) and `tests/auth/dual-verify-dispatch.test.js` (10 cases: segment-count routing, malformed input rejection, strict no-fall-through dispatch, legacy verifier regressions). Existing `tests/auth-token-validation.test.js` continues to pass via the legacy HMAC path — acts as a free legacy-compat regression test. All 26 new tests green on first run; typecheck + lint clean.
- **Lesson 1 — Stale specs hide test rot.** When a function's contract widens (here: `requireAuth` gained DB-backed session enforcement on 2026-01-05), every test that exercises it gains a hidden invariant. The pre-existing `tests/auth-token-validation.test.js` "extracts userId from valid token" test was written for a pre-2026-01-05 `requireAuth` and became stale when session lookup was added — but the test still ran, just permanently red. **When a function's contract widens, its test suite must widen too, or the tests become a misleading green/red signal.** Logged as D-105 in DOC_DISCREPANCIES for separate fix.
- **Lesson 2 — Pinned algorithms list is structural defense against confusion attacks.** `jwtVerify(token, key, { algorithms: ['HS256'] })` is non-negotiable. Without it, jose accepts whatever algorithm the token header declares — meaning an attacker could craft an `alg: none` token (no signature) and have it verified. The pinned list is structural enforcement at the JWT-verify layer, not an optional hardening flag.
- **Lesson 3 — Dual-verify rollout via segment-count dispatch is itself the feature flag.** A 3-segment token IS a JWT, a 2-segment token IS legacy HMAC — no separate runtime flag needed, no four-state matrix (flag on/off × format new/old). The cleanup PR becomes mechanical (delete one branch in a switch) rather than coordinated (flip flag, then delete code, then delete flag). Same "structural enforcement over discipline" principle as the case-collision lint from PR #29.
- **Lesson 4 — Variable-name preservation minimizes diff size during refactor.** When consolidating the 53-line inline verifier in `location.js`, keeping the variable name `authenticatedUserId` (vs. switching to `req.auth.userId` everywhere) avoided touching 9 other downstream code points. Result: 53 lines removed → 1 line replacement, with zero downstream changes. The "minimal diff" principle: a refactor's blast radius should match the actual logic change, not multiply because of incidental renames.

## 2026-05-02: Logger taxonomy migrations — spec the exceptions, don't violate them

- **Symptom:** Tier 3 of the matrixLog migration (auth.js + location.js + blocks-fast.js) needed to keep the existing `userId.substring(0, 8)` correlation prefix in auth-flow log messages for real-time DX, but the matrixLog spec at `server/logger/matrix.js:26-28` (written 2026-05-01) explicitly forbade UUIDs in messages. Initial Claude recommendation was to drop the prefix entirely (Option (i), spec-literal); deeper review revealed this would degrade Replit console correlation when multiple drivers log in concurrently.
- **Root cause:** A new structured logging spec was written on top of an existing convention without auditing the convention first. The auth-flow `userId.substring(0, 8)` pattern (`auth.js:56` and similar) predated the matrixLog spec by months and was load-bearing for debuggability — but the spec text was authored as if no prior pattern existed.
- **Resolution (Melody, 2026-05-02):** modified Option (ii) — keep the 8-char hex prefix in messages, AND amend the matrixLog spec to explicitly permit it as a deliberate exception. Spec-amendment commit (`docs(matrix): permit 8-char hex correlation prefix in messages`) lands BEFORE the migration commits so the spec authorizes the convention rather than being silently violated. Email / coord / full-address / full-UUID remain forbidden in messages.
- **Lesson 1 — Audit existing conventions before writing fresh specs.** When introducing a new structured logging spec on top of existing convention, grep the active convention first. Either (a) explicitly carry it forward into the spec as a permitted exception, or (b) deprecate it deliberately. "We'll fix it during migration" rationalization leads to silent spec violations.
- **Lesson 2 — Spec-amendment commits should precede the migration commits that rely on them.** The git timeline reads as *spec said X → spec said Y → code does Y*. If migration commits land first with the convention, future readers grep the spec, see "NO UUIDs in messages", and assume the code is buggy. By landing the spec amendment first, the timeline is self-explanatory.
- **Lesson 3 — Real-time DX is a real architectural concern.** Spec-literal compliance can be the right call for log-pipeline ingest, but the cost of "you must tail JSON sidecar to correlate" should be measured against the value, not assumed away. ## 2026-04-30: Hash Function Normalization Gap → 161 Duplicate Events Past UNIQUE Constraint

- **Symptom:** `discovered_events` accumulated 161 dupes (out of 1826 rows) despite `event_hash` having a `UNIQUE` constraint at the schema level. Read-path application logic ran `deduplicateEvents` + `deduplicateEventsSemantic` ~15× per page load on `/co-pilot/strategy` to compensate, masking the underlying integrity gap.
- **Root Cause:** `generateEventHash` (write-side) used narrower title normalization than `deduplicateEvents` (read-side). The hash stripped only "at Venue" suffixes; the dedup also stripped prefixes ("Live Music:", "Concert:", "Show:"), parentheticals ("(Shared Reality)"), and free-form suffixes. Same logical event with title variations therefore produced different hashes, bypassing the UNIQUE constraint.
- **Aggravator:** Read-path dedup ran in 4 places (3 in `briefing.js`, 1 post-SELECT inside `fetchEventsForBriefing` in `briefing-service.js`). The noise was familiar; the underlying gap wasn't surfaced for months.
- **Initial proposal (rejected):** Add a second UNIQUE on `(state, lower(title), event_start_date, venue_id)` as a composite key. Melody rejected: `lower(title)` requires exact match → would force title-mangling in the DB to catch prefix variations, degrading UI presentation.
- **Actual fix (Choice A):** Upgrade `generateEventHash`'s normalization to match `deduplicateEvents` (strip prefixes, parentheticals; extract normalized street name from address). Migrate existing 1826 rows by re-hashing with the new logic in a Node script that imports the same function the app uses → guaranteed parity. Strip dedup from all 4 read sites; relocate to PRE-INSERT in the briefing pipeline. Result: 161 dupes consolidated; 0 dupes by `event_hash` post-migration; read-path silent on DEDUP under `[API]` chain; runtime logs confirmed `[BRIEFING] [EVENTS] [DEDUP] [WRITE] hash: 15 → 15, semantic: 15 → 15` firing once per briefing generation.
- **Files:** `server/lib/events/pipeline/hashEvent.js` (v3), `server/scripts/migrate-event-hashes.js`, `server/lib/briefing/briefing-service.js`, `server/api/briefing/briefing.js`, `tests/events/pipeline.test.js` (12 new Choice A unit tests).
- **Lesson 1 — Late-binding identity beats early-binding constraints when normalization needs to evolve.** A computed hash is *late-binding*: the function lives in code, can change, and decouples identity from presentation. A composite-key index is *early-binding*: it locks identity to literal column values. The hash-function approach lets normalization evolve without schema migration — only data migration via a Node script that imports the same function.
- **Lesson 2 — Audit findings should update the recommendation, not be rationalized into the original framing.** The plan was framed as "add a constraint" before the audit. The audit revealed a constraint already existed. The right response was to redesign from first principles (Choice A). Anchoring on plan-aligned options is its own failure mode — when audit findings change the problem statement, the recommendation should change too.
- **Lesson 3 — `[Action]` chains in logs are stage specifications, not readability niceties.** A log line showing `[BRIEFING] [API] [EVENTS] [DEDUP]` was structural evidence that DEDUP was running at the wrong stage. The right response is to relocate the operation to its correct waterfall stage (write, not read) — not to memoize/cache/suppress the noise. Per claude_memory rows 268–269.

- **Plan ref:** `docs/review-queue/PLAN_events-dedup-architectural-2026-04-30.md` (Course Correction Log + Decision Log + 8 test cases).

## 2026-02-13: Logout Race Condition — Cancel Queries Before Clearing Auth Token

- **Symptom:** Clicking logout redirected to sign-in page, but immediately showed the red FAIL HARD (CriticalError) screen before the user could log back in.
- **Root Cause:** React Query keeps background queries running independently of component lifecycle. When `logout()` cleared the auth token, React didn't instantly unmount the protected `CoPilotContext` and `GlobalHeader` components. During that render-cycle delay, in-flight queries (location polling, strategy fetching) fired with the now-invalid token, received 401 responses, and triggered `setCriticalError()` — the blocking error modal.
- **Fix:** Call `queryClient.cancelQueries()` and `queryClient.clear()` at the **top** of `logout()`, BEFORE clearing the token or calling the server. This aborts all in-flight HTTP requests immediately so no stale 401 callbacks can fire.
- **File:** `client/src/contexts/auth-context.tsx`
- **Lesson:** Any app with background polling behind authentication MUST cancel all active queries as the first step of logout. The order matters: cancel queries → call server → clear local state. If you clear the token first, every in-flight request becomes an error.
